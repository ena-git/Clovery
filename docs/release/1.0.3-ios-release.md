## Scope

- Preserve the existing bundle IDs, App Group, iCloud container, user data, and
  App Store listing.
- Include the V1 photo-library save fix and StoreKit entitlement recovery fix.
- Include migration export support required by the later Clovery-account/Vault
  transition.
- Keep the Go backend and Flutter multi-platform work in the repository without
  using the Flutter app as this App Store binary.

## Release Flow

1. Merge the source release pull request into `main` without rewriting history,
   then record that exact source candidate commit SHA.
2. From that source candidate commit, build
   `build/Clovery-1.0.3-14.xcarchive`, upload TestFlight build `14`, and complete
   the real-device, migration, photo-library, Sandbox, and StoreKit acceptance
   gates. Export `CLOVERY_RELEASE_COMMIT=<source-candidate-sha>` before the
   Archive command and pass
   `CLOVERY_SOURCE_COMMIT="$CLOVERY_RELEASE_COMMIT"` to `xcodebuild archive`;
   the signed app must carry that exact source SHA.
3. Create an evidence-only pull request. It may update only
   `docs/release/*.md` and must contain only non-sensitive acceptance summaries.
4. Merge the evidence pull request, set
   `CLOVERY_RELEASE_COMMIT=<source-candidate-sha>`, and run
   `scripts/verify-ios-release-evidence.sh`. The checker must confirm that only
   release Markdown changed from the source candidate through current `HEAD` and
   that the local signed Archive exists.
5. After the checker passes, associate TestFlight build `14` with App Store
   version `1.0.3` and the existing non-consumable product
   `com.clovery.app.board.lifetime`, then submit them for review. Confirm that
   App Store Connect has accepted the submission into review, that its source
   matches the recorded candidate, and that current `main` differs from that
   candidate only by release Markdown. Only then create tag `ios-v1.0.3` on
   current `main` and publish the GitHub release.

Flutter work remains blocked until App Store Connect accepts the native iOS
submission into review and tag `ios-v1.0.3` points to the same current `main`
commit that passed this release flow.

Native CI and the signed App Store Archive use non-beta Xcode `26.0.1`
(`17A400`). The workflow pins the GitHub runner's corresponding Xcode path so a
runner-default change cannot silently alter the release toolchain. Automated
XCTest selects the highest available iOS simulator runtime unless an available
iPhone ID is explicitly supplied for compatibility testing.

## Manual Evidence Gate

- Automated CI does not call `scripts/verify-ios-release-evidence.sh`. Run this
  manual gate only after the evidence-only pull request is merged into `main`
  and before the App Store Connect review submission.
- The checker trusts only the committed
  `HEAD:docs/release/ios-1.0.3-acceptance.md`; uncommitted working-tree edits are
  never release evidence. It also verifies the Archive metadata, iPhoneOS
  platform, arm64 architecture, source SHA, versions, bundle identifiers, Team
  `M92TBSSR2R`, required entitlements, executables, and strict code signatures
  for the app and widget.
- No one may mechanically replace `NOT_RUN` with `PASS`. Every passing item must
  use this exact completion format:
  `- [x] <exact label> — PASS — Time: <nonempty>; Device: <nonempty>; OS: <nonempty>; Evidence: <nonempty>`
  Populate it with the real test time, device model, operating-system version,
  and a non-sensitive evidence filename.
- Raw counts, hashes, screenshots, logs, and archives must not enter Git. Keep
  them only in the gitignored `build/release-evidence/ios-1.0.3/` directory.

Build the candidate Archive with the source binding enabled:

```bash
export CLOVERY_RELEASE_COMMIT="$(git rev-parse HEAD)"
xcodebuild \
  -project Clovery.xcodeproj \
  -scheme Clovery \
  -configuration Release \
  -destination 'generic/platform=iOS' \
  -archivePath build/Clovery-1.0.3-14.xcarchive \
  CLOVERY_SOURCE_COMMIT="$CLOVERY_RELEASE_COMMIT" \
  archive
```
