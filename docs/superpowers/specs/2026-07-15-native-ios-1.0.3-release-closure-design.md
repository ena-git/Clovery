# Clovery 原生 iOS 1.0.3 发布收口设计

**状态：** 已批准方案 A
**目标版本：** `1.0.3 (14)`
**现网基线：** `1.0.2 (13)`
**App Bundle ID：** `com.clovery.app`
**终身权益商品：** `com.clovery.app.board.lifetime`
**计划 Git 标签：** `ios-v1.0.3`

## 1. 决策

在开始 Flutter 跨端客户端开发前，先完成并发布现有原生 iOS 应用的维护升级。此次收口优先解决已经影响现网用户的数据安全、照片保存和付费权益问题，并为后续 V1 数据迁移到 Clovery 根账户与 Vault 做好只读、可校验的出口。

本设计只调整总控计划的执行顺序，不改变既有技术方向：

1. 先完成原生 iOS `1.0.3 (14)` 的代码、真机、StoreKit、TestFlight 和升级发布门禁。
2. 原生 iOS 发布完成并保留验收证据后，才开始 Flutter iOS 垂直切片。
3. Flutter iOS 达到独立 Beta 门槛后，才开始 Android 与 HarmonyOS 工作流。
4. Go 后端继续使用 Clovery 根账户与 Vault 模型；Apple、Google、Huawei、微信和 QQ 仅作为可绑定登录身份。

本设计在执行顺序上优先于 `2026-07-10-clovery-rebuild-program.md` 和 `2026-07-10-w1-v1-data-protection.md` 中“W3/W4 后再做原生 iOS 真机验收”的旧排期描述。后续实施会同步修正文档，避免两套顺序并存。

## 2. 范围边界

### 2.1 本次必须完成

- 保持当前 App Store 应用的 Bundle ID、App Group、iCloud 容器、签名能力和升级关系不变。
- 修复并验证“照片只能分享、不能保存到系统相册”的 P0 问题。
- 修复并验证购买后未解锁、恢复购买失败或启动后权益丢失的 P0 问题。
- 验证升级安装不会丢失日记、照片、删除墓碑和已保存权益状态。
- 验证迁移 Bundle 可以重复导出、完整校验且不会修改 V1 原数据。
- 完成 App Store Connect 商品配置核对、Sandbox 购买矩阵和 TestFlight 冒烟。
- 形成可追溯的发布证据，合并到 `main` 后再创建标签和 GitHub Release。

### 2.2 本次不做

- 不在旧 WebView 应用中新增 Clovery ID 注册、密码重置、Passkey 或第三方身份绑定页面。
- 不把旧 V1 数据直接写入 Go 后端或 V2 Vault。
- 不把 Flutter 工程作为 `1.0.3 (14)` 的 App Store 二进制。
- 不开始 Android 或 HarmonyOS 功能实现。
- 不重写旧数据格式，不自动删除旧照片、备份或迁移 Bundle。

## 3. 架构一致性

### 3.1 V1 兼容层

原生 iOS `1.0.3` 继续作为现有用户的兼容维护版本：Swift/SwiftUI 提供系统能力，WKWebView 承载现有日记界面，`PhotoStore` 管理 Documents 照片，现有 localStorage、KVS 和 CloudKit 数据保持可读。此次版本只做向后兼容修复和只读迁移导出，不引入破坏性数据迁移。

### 3.2 V2 根账户边界

V2 的稳定模型保持不变：

```text
Clovery Account
└── vault_id
    ├── 日记与删除墓碑
    ├── 图片对象与同步记录
    ├── 云端配额
    └── 权益与订阅
```

设备不是数据根，Apple、Google、Huawei、微信和 QQ 也不是账户根。V1 没有 Clovery 根账户，因此此次维护版不能伪造账户归属，也不能把设备标识升级为账户标识。V1 通过版本化迁移 Bundle 把数据安全地交给未来 Flutter/V2 的显式导入流程。

### 3.3 权益边界

`1.0.3` 的现网终身权益仍由 Apple StoreKit 商品 `com.clovery.app.board.lifetime` 决定。客户端只接受已验证交易，并在启动、购买更新和“恢复购买”时重新读取当前有效权益。V2 上线后，服务端可以把经 Apple 验证的权益归属到 `clovery_account_id`；本次不在无根账户的 V1 中提前建立不可靠映射。

## 4. 模块与职责

### 4.1 照片存储与系统相册

- `PhotoStore` 只负责应用 Documents 照片的原子保存、读取和垃圾回收。
- WebView 照片桥只负责 JavaScript 与 `PhotoStore` 之间的结构化请求和错误回传。
- 系统相册保存使用 iOS Photos 能力，独立于系统分享面板；分享成功不能被视为保存到相册成功。
- 权限状态必须区分未决定、允许、受限和拒绝。拒绝时提供明确说明与进入系统设置的恢复入口。
- 只有 Photos 框架确认写入成功后，界面才显示“已保存到相册”。

### 4.2 StoreKit 权益

- `BoardStore` 负责商品查询、购买、恢复、交易监听和当前权益刷新。
- 购买结果固定区分 `success`、`cancelled`、`pending` 和 `failed`，不得把取消或待批准误报为失败或成功。
- 只使用通过 StoreKit 验证的交易解锁权益；未验证交易必须记录诊断信息但不能解锁。
- App 启动、回到前台、交易更新和用户点击恢复购买时都必须刷新当前权益。
- 已验证且已经交付的交易应完成 `finish()`，防止重复投递；完成前必须先更新本地权益状态。
- WebView 只接收结构化权益状态，不直接判断交易真伪。

### 4.3 升级与迁移导出

- 升级安装必须复用现有 App 容器、App Group 和 iCloud 容器，不创建新的数据根目录。
- 迁移导出只读取日记 JSON、删除墓碑、Documents 照片、可用备份与 CloudKit 数据。
- 每次导出使用独立 `migration_id`，包含固定格式版本、条目计数、照片计数和 SHA-256 清单。
- 导出失败不能覆盖上一次成功结果，导出成功也不能删除或改写 V1 原数据。

### 4.4 发布配置

- 主 App 和 Widget 必须同时报告 `1.0.3 (14)`。
- Release Archive 必须保留 `com.clovery.app`、`group.com.clovery.app` 和当前 iCloud 容器能力。
- `Info.plist` 必须包含与实际功能一致的相册写入权限文案。
- App Store Connect 中的终身商品类型、价格、本地化、销售地区、审核截图和版本关联必须完整。

## 5. 关键数据流

### 5.1 保存到系统相册

```text
用户点击保存
→ WebView 生成可导出的图片数据
→ 原生桥校验数据与文件类型
→ 请求或检查 Photos 写入权限
→ Photos 执行资源创建
→ 成功：结构化回调“已保存”
→ 失败：返回可分类错误和恢复操作
```

分享面板属于另一条数据流，不得替代 Photos 写入，也不得共享模糊的成功状态。

### 5.2 购买与恢复权益

```text
启动或恢复前台
→ 查询 StoreKit 当前有效权益
→ 验证交易
→ 更新原生权益状态
→ 结构化回调 WebView

用户购买
→ Product.purchase()
→ success：验证交易 → 更新权益 → finish
→ pending：保持未解锁并等待交易监听
→ cancelled：保持原状态
→ failed：显示可重试错误
```

### 5.3 V1 到 V2 的未来迁移

```text
V1 只读导出 migration_bundle.zip
→ 用户在未来 Flutter/V2 中显式导入
→ 客户端校验格式、数量和 SHA-256
→ 后端在已认证的 clovery_account_id 下创建迁移任务
→ 数据归属到该账户的 vault_id
→ V1 原数据保留至迁移验收完成
```

## 6. 错误处理与可观测性

- 用户可恢复错误必须提供明确操作：重试、打开系统设置、恢复购买或稍后再试。
- 权限拒绝、Photos 写入失败、商品不可用、未验证交易和迁移校验失败必须使用稳定错误代码，不依赖解析中文提示。
- 日志不得包含日记正文、照片 base64、购买凭据或用户敏感信息。
- StoreKit 诊断至少记录环境、商品 ID、交易状态和验证结果；不记录可重放凭据。
- 每个发布阻塞项必须记录测试版本、设备、系统版本、时间和结果；失败项不能仅凭口头确认关闭。

## 7. 验收工作流

### 7.1 自动化与构建

- XCTest 覆盖照片存储、WebView 桥、StoreKit 回调和迁移导出。
- HTML/Babel 校验和 P0 合同测试通过。
- Debug 模拟器构建、Release iPhoneOS Archive 和签名能力检查通过。
- 主 App 与 Widget 的版本号、Bundle ID、App Group 和 iCloud entitlement 一致。

### 7.2 升级与数据安全真机验收

- 在安装 `1.0.2 (13)` 且包含日记、照片、删除墓碑和权益样本的设备上覆盖安装 `1.0.3 (14)`。
- 核对首页、日历、详情、分享卡、重启后状态和 CloudKit 拉取结果。
- 导出迁移 Bundle，核对条目数、删除数、照片数和全部 SHA-256，随机打开至少三张照片。
- 重复导出后确认两个迁移目录均保留，V1 原数据完全不变。

### 7.3 照片真机验收

- 首次授权后保存成功并能在系统相册打开。
- 已授权状态下重复保存成功。
- 拒绝权限时不误报成功，并能通过系统设置恢复。
- 受限权限、写入失败和应用重启场景有明确反馈。
- 保存、分享和应用内照片持久化分别验证，三者不能互相替代。

### 7.4 StoreKit Sandbox/TestFlight 验收

- 查询并显示 App Store Connect 的真实本地化价格。
- 取消购买不解锁，Ask to Buy pending 不解锁。
- Sandbox 成功购买立即解锁，杀掉 App 后仍能恢复。
- 重装应用和第二台设备点击恢复购买后重新解锁。
- App Store Connect 能确认商品配置和相应测试/生产交易来源；若销售趋势页面尚未归集，以交易和财务报告的环境、时区与结算延迟说明为准，不把单张趋势截图作为唯一判断依据。

### 7.5 TestFlight 与发布

- TestFlight 完成启动、日记、照片、相册保存、分享、导出、购买和恢复购买冒烟。
- 所有阻塞门禁有证据并在发布清单勾选完成。
- 发布提交先合并到 `main`，再基于实际上传并提交审核的提交创建 `ios-v1.0.3`。
- GitHub Release 与 App Store 版本说明使用同一修复范围，不承诺尚未上线的 Flutter 或跨端同步功能。

## 8. 发布阻断规则

出现以下任一情况时不得提交 App Store 审核或发布 GitHub Release：

- 任一自动化测试、签名 Archive 或版本一致性检查失败。
- 升级安装后出现日记、照片、墓碑或权益丢失。
- 系统相册保存只能通过分享面板间接完成，或权限拒绝后无法恢复。
- 已验证 Sandbox 购买不能立即解锁，或恢复购买在重装/第二设备失败。
- App Store Connect 商品未达到可随版本提交的状态。
- TestFlight 冒烟未完成，或验收证据缺少设备、版本和结果。

## 9. 回滚策略

- App Store 提交前发现问题：停止上传或撤回候选构建，修复后递增 build number，不能覆盖已上传构建。
- TestFlight 发现数据风险：立即停止外部测试分发，保留失败设备和迁移 Bundle 作为证据。
- App Store 发布后发现 P0：暂停分阶段发布；若已全量发布，准备不破坏数据格式的热修复 build。
- 任何回滚都不得删除 V1 用户数据、迁移导出或现有 CloudKit 记录。

## 10. Flutter 启动门槛

只有满足以下条件，才允许把主执行计划切换到 Flutter：

1. 原生 iOS `1.0.3` 的全部发布阻塞项通过并留存证据。
2. App Store Connect 已接受候选构建进入审核，或版本已经发布；若审核被拒，必须先解决与当前版本有关的问题。
3. `main` 包含已提交审核的源代码，`ios-v1.0.3` 指向同一发布提交。
4. V1 迁移 Bundle 格式冻结，Flutter 导入端可以把它作为稳定输入。
5. Flutter 工作流继续遵守 `clovery_account_id` 与 `vault_id` 根模型，不复用 V1 的设备身份或 Apple-only 存储作为跨端核心。

达到上述门槛后，执行顺序恢复为 Flutter iOS 核心 → 迁移与同步 Beta → iOS V2 真机/TestFlight → Android → HarmonyOS。
